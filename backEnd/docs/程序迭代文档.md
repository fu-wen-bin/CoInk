# 2026.3.6 GitHub 登录 Token 验证修复

## 问题描述

GitHub OAuth 登录回调显示 `status=ok`，但提示 "Token 验证失败"。

## 根因分析

前端 `useGitHubCallback` hook 中错误地判断了 `verifyToken` 的响应格式：

- 实际返回：`{ code: 200, data: { valid: true, payload: {...} } }`（标准 ApiResponse 包装）
- 错误判断：代码检查了 `verifyData.code !== 200`，但 `verifyToken` 的返回类型是 `ApiResponse<VerifyTokenResponseData>`
  ，需要通过 `verifyData.data.valid` 判断
- 此外，在严重JWT的时候传递的参数是`Bearer ${accessToken}`，而不是`accessToken`变量参数，导致验证失败。

## 修复内容

**修改 useGitHubCallback hook** (`coInk/src/hooks/useAuth.ts`)

- 修正 `verifyToken` 响应判断逻辑：`verifyData.code !== 200 || !verifyData.data?.valid`
- 修正 payload 获取方式：`verifyData.data.payload`
- 修正 token 传递方式：`this.authService.verifyToken(accessToken)`

## 修改文件

- `coInk/src/hooks/useAuth.ts`

# 2026.3.7 退出登录功能完善 - 后端 Token 清理

## 问题描述

用户点击退出登录时，仅清除了前端的登录标记（`logged_in` Cookie）和 localStorage 中的用户信息，但未调用后端接口清除 HTTP-Only
Cookie 中的双 token（`access_token` 和 `refresh_token`），导致 token 仍然存在安全隐患。

## 根因分析

1. **前端退出逻辑不完整**：仅清除了前端可见的状态，未通知后端清除 HTTP-Only Cookie
2. **缺少复用逻辑**：多个组件可能需要退出登录功能，但各自实现会导致代码重复
3. **跳转方式不统一**：部分代码使用 `window.location.href`，部分使用 Next.js router

## 修复内容

### 1. 完善 useLogoutMutation Hook (`coInk/src/hooks/useUserQuery.ts`)

- 导入 Next.js `useRouter` 替代 `window.location.href`
- 明确注释说明完整登出流程：
    1. 调用后端 `/auth/logout` 接口清除 HTTP-Only Cookie 中的双 token
    2. 清理 React Query 缓存
    3. 清理 localStorage 中的用户信息
    4. 清理前端登录标记 Cookie
    5. 使用 router 跳转到登录页

```typescript
export function useLogoutMutation () {
  const queryClient = useQueryClient();
  const router = useRouter();

  return useMutation({
    mutationFn: () => authApi.logout(), // 调用后端接口清除 Cookie
    onSuccess: () => {
      queryClient.removeQueries({ queryKey: userQueryKeys.all });
      storage.clear();
      clearLoggedInFlag();
      router.push('/auth'); // Next.js 路由跳转
    },
  });
}
```

### 2. 首页使用 useLogoutMutation (`coInk/src/app/page.tsx`)

- 导入 `useLogoutMutation` hook
- 移除本地实现的 `clearUserFromStorage` 和 `USER_STORAGE_KEY`（已封装在 hook 中）
- 简化 `handleLogout` 函数，直接调用 `logoutMutation.mutate()`
- 添加 `useEffect` 监听登出成功状态，更新本地 `isLoggedInState`

### 3. 新增通用退出登录工具函数 (`coInk/src/utils/auth/logout.ts`)

创建独立的退出登录工具模块，供非 React 环境使用：

- **`logout(options)`** - 异步完整登出
    - 调用后端 `/auth/logout` 接口
    - 清理所有前端状态
    - 支持 `redirectTo`、`onSuccess`、`onError` 配置

- **`logoutSync(options)`** - 同步快速登出
    - 仅清理前端状态，不调用后端
    - 用于紧急清理或后端已失效场景

- **`shouldRefreshToken()`** - Token 刷新检查占位
    - HTTP-Only Cookie 方案下前端无法直接判断，由拦截器处理

```typescript
// 使用示例
import { logout } from '@/utils/auth/logout';

// 基础使用
await logout();

// 带跳转
await logout({ redirectTo: '/auth' });

// 带回调
await logout({
  onSuccess: () => console.log('登出成功'),
  onError: (error) => console.error('登出失败', error)
});
```

## 后端接口说明

后端 `/auth/logout` 接口（`POST`）：

- 清除 HTTP-Only Cookie 中的 `access_token`
- 清除 HTTP-Only Cookie 中的 `refresh_token`
- 返回 `{ code: 200, message: '退出成功' }`

- `coInk/src/app/page.tsx` - 使用 useLogoutMutation，简化退出逻辑
- **新增** `coInk/src/utils/auth/logout.ts` - 通用退出登录工具函数

```

调用 authApi.logout() → POST /auth/logout
    ↓
后端清除 HTTP-Only Cookie (access_token, refresh_token)

    ↓
前端清理 localStorage (cached_user_profile)
```

# 2026.3.7 Dashboard 页面水合（Hydration）错误修复

## 问题描述

登录后刷新 Dashboard 页面时，Next.js 报告水合错误：

```
Hydration failed because the server rendered HTML didn't match the client.
```

具体表现为：

- 服务端渲染 `<div>`（用户未加载时的骨架屏）
- 客户端期望 `<button>`（用户加载后的下拉菜单触发器）
- 错误发生在 `DashboardHeader` 组件的用户区域

## 根因分析

**服务端 vs 客户端数据不一致**：

- **服务端**：`displayUser` 为 `undefined`（localStorage 和 React Query 缓存在服务端不可用）
- **客户端**：`displayUser` 从 localStorage 恢复，有值

导致条件渲染结果不同：

```tsx
{
  displayUser ? (
    <DropdownMenuTrigger asChild>
      <Button>...</Button> {/* 客户端渲染 */}
    </DropdownMenuTrigger>
  ) : (
    <div>...</div>
  {/* 服务端渲染骨架屏 */}
)
}
```

## 修复内容

### 修改 DashboardHeader 组件 (`coInk/src/components/dashboard/DashboardHeader.tsx`)

使用 `mounted` 状态延迟用户区域的渲染，确保 SSR 和客户端初始渲染一致：

```tsx
// 1. 添加挂载状态检查（第 69-71 行已存在）
useEffect(() => {
  setMounted(true);
}, []);

// 2. 修改条件渲染，添加 mounted 检查
{
  mounted && displayUser ? (
    // 已登录用户下拉菜单
    <DropdownMenuTrigger asChild>
      <Button>...</Button>
    </DropdownMenuTrigger>
  ) : mounted ? (
    // 客户端挂载后但无用户数据 - 骨架屏
    <div>...</div>
  ) : (
    // SSR 时渲染占位符，保持与客户端初始渲染一致
    <div className="h-11 w-11 rounded-xl bg-gradient-to-br from-gray-50 to-gray-100" />
  )
}
```

## 修复要点

1. **延迟条件渲染直到客户端挂载完成**
    - 使用 `mounted` 状态控制渲染
    - SSR 时始终渲染固定占位符
    - 客户端挂载后再根据 `displayUser` 渲染实际内容

2. **确保占位符结构一致**
    - SSR 和客户端首次渲染都使用 `<div>` 占位符
    - 避免因数据结构变化导致的 HTML 不匹配

## 修改文件

- `coInk/src/components/dashboard/DashboardHeader.tsx` - 添加 mounted 检查，修复水合错误
- `coInk/src/components/notifications/notification-dropdown.tsx` - 添加 mounted 检查，修复 Radix UI id 不匹配

## 额外修复：NotificationDropdown 水合错误

### 问题

NotificationDropdown 组件也报告了类似的水合错误，原因是 Radix UI 的 DropdownMenu 在服务端和客户端生成不同的 `id` 属性（如
`radix-_R_77cndlb_` vs `radix-_R_stkndlb_`）。

### 修复

在 NotificationDropdown 中添加 `mounted` 状态，SSR 时渲染禁用状态的占位 Button，客户端挂载后再渲染完整的 DropdownMenu：

```tsx
const [mounted, setMounted] = useState(false);

useEffect(() => {
  setMounted(true);
}, []);

// SSR 时渲染占位符
if (!mounted) {
  return (
    <Button variant="ghost" size="sm" disabled>
      <Bell className="h-5 w-5 text-gray-600" />
    </Button>
  );
}

// 客户端挂载后渲染完整组件
return (
  <DropdownMenu>...</DropdownMenu>
);
```

## 最佳实践总结

避免 Next.js 水合错误的常用方法：

1. **使用 mounted 状态延迟渲染** - 适用于依赖浏览器 API 的组件
2. **确保 SSR/CSR 渲染一致** - 初始渲染不依赖会变化的数据
3. **使用 suppressHydrationWarning** - 仅用于时间戳等不可避免的微小差异
4. **动态导入禁用 SSR** - 对纯客户端组件使用 `dynamic(() => import(...), { ssr: false })`

# 2026.3.7 邮箱密码登录用户信息缓存修复

## 问题描述

用户使用邮箱密码登录后，右上角无法显示用户信息，localStorage 也没有保存用户数据。但 GitHub 登录功能是正常的。

## 根因分析

React Query 的查询 key 不匹配问题：

- `handleAuthSuccess` 设置缓存时使用的 key：`['user', 'profile']`
- `useUserQuery` hook 查询时使用的 key：`['user', 'profile', undefined]`（因为不传 userId）

在 React Query 中，这两个 key 是不同的：

```javascript
['user', 'profile'] !== ['user', 'profile', undefined]
// 长度不同，被认为是不同的 key
```

这导致：

1. 登录成功后 `handleAuthSuccess` 将用户数据设置到 key `['user', 'profile']`
2. `DashboardHeader` 调用 `useUserQuery()`，它从 key `['user', 'profile', undefined]` 读取
3. 查询不到数据，显示为无用户状态

## 修复内容

### 1. 修复 handleAuthSuccess (`coInk/src/hooks/useAuth.ts`)

使用与 `useUserQuery` 相同的查询 key 格式：

```typescript
// 修复前
queryClient.setQueryData<User>(userQueryKeys.profile(), user);

// 修复后 - 与 useUserQuery 使用的 key 格式一致
queryClient.setQueryData<User>([...userQueryKeys.profile(), undefined], user);
```

### 2. 修复 getLocalUserData (`coInk/src/hooks/useUserQuery.ts`)

同样更新此函数中的查询 key：

```typescript
// 修复前
return queryClient.getQueryData<User>(userQueryKeys.profile()) ?? storage.get() ?? undefined;

// 修复后 - 使用正确的 key 格式
return queryClient.getQueryData<User>([...userQueryKeys.profile(), undefined]) ?? storage.get() ?? undefined;
```

### 3. 关键修复：auth/page.tsx 登录数据保存

**问题**：之前的修复只处理了 `handleAuthSuccess` 函数，但 `auth/page.tsx` **根本没有调用这个函数**！它直接调用
API，登录成功后只设置了 `setLoggedInFlag()` 就跳转了。

**修复**：在 `auth/page.tsx` 中：

1. 导入 `useQueryClient`、`userQueryKeys` 和 `User` 类型
2. 添加 `saveUserToStorage` 辅助函数
3. 在登录成功后保存用户数据到 React Query 缓存和 localStorage
4. 同样在注册成功后保存用户数据

```typescript
// 新增导入
import { useQueryClient } from '@tanstack/react-query';
import { userQueryKeys } from '@/hooks/useUserQuery';
import type { User } from '@/services/auth/types';

// 登录成功后保存用户数据
if (authData?.user) {
  queryClient.setQueryData<User>([...userQueryKeys.profile(), undefined], authData.user);
  saveUserToStorage(authData.user);
}
```

## 问题根源

`useUserQuery` 的定义：

```typescript
export function useUserQuery (userId?: string) {
  return useQuery({
    queryKey: [...userQueryKeys.profile(), userId], // 包含 userId 参数
    // ...
  });
}
```

当调用 `useUserQuery()` 不传参数时，`userId` 是 `undefined`，所以实际 key 是 `['user', 'profile', undefined]`。

但其他地方的代码假设 key 是 `userQueryKeys.profile()`（即 `['user', 'profile']`），导致不匹配。

## 最佳实践

定义 query key 时应注意一致性：

```typescript
// 推荐：统一定义所有可能的 key 变体
export const userQueryKeys = {
  all: ['user'] as const,
  profile: () => [...userQueryKeys.all, 'profile'] as const,
  profileById: (userId: string | undefined) => [...userQueryKeys.profile(), userId] as const,
};

// 使用
useQuery({ queryKey: userQueryKeys.profileById(userId), ... })
queryClient.setQueryData(userQueryKeys.profileById(undefined), data)
```

## 修改文件

- `coInk/src/hooks/useAuth.ts` - 修复 handleAuthSuccess 中的查询 key

# 2026.3.9 修复 dashboard/user 目录下 layout 与 page 职责混淆

## 问题描述

当前 `/dashboard/user/` 目录结构存在职责不清的问题：

1. **数据获取重复**：`layout.tsx` 和 `page.tsx` 都调用 `useUserQuery()` 获取用户数据
2. **骨架屏重复**：layout 有完整的页面骨架，page 也有表单骨架，造成视觉冲突
3. **错误处理位置不当**：page.tsx 处理了全局错误状态，但布局层也需要用户数据

## 根因分析

Next.js 嵌套布局架构中：

- **Layout** 应该负责共享 UI 和共享数据获取
- **Page** 应该只负责特定路由的内容

但当前实现中，page.tsx 重复获取了 layout 已经获取的数据，且两者都有骨架屏和错误处理逻辑，导致：

1. 同一数据被请求两次（虽然 React Query 缓存可以避免重复网络请求，但代码逻辑混乱）
2. 用户可能看到两个骨架屏叠加
3. 错误处理分散在不同层级

## 修复内容

### 1. 新增 UserProfileContext (`coInk/src/app/dashboard/user/_components/user-profile-context.tsx`)

创建 Context 用于在 layout 和 page 之间共享用户数据：

```typescript
interface UserProfileContextType {
  user: User | undefined;
  isLoading: boolean;
  error: Error | null;
}

export function UserProfileProvider({ children, user, isLoading, error }: UserProfileProviderProps) {
  return (
    <UserProfileContext.Provider value={{ user, isLoading, error }}>
      {children}
    </UserProfileContext.Provider>
  );
}

export function useUserProfile() {
  const context = useContext(UserProfileContext);
  if (context === undefined) {
    throw new Error('useUserProfile must be used within a UserProfileProvider');
  }
  return context;
}
```

### 2. 更新 layout.tsx

**职责**：共享 UI + 数据获取 + 全局状态处理

- 保留用户数据获取（用于头部显示）
- 保留骨架屏（全局加载状态）
- 新增错误处理组件 `UserLayoutError`
- 使用 `UserProfileProvider` 包裹 children，传递用户数据

```typescript
// 获取用户数据
const { data: profile, isLoading, error, refetch } = useUserQuery();

// 错误状态处理
if (error && !displayProfile) {
  return <UserLayoutError error={error} onRetry={refetch} />;
}

// 通过 Provider 共享数据给子页面
return (
  <UserProfileProvider user={displayProfile} isLoading={isLoading} error={error}>
    {/* 头部、导航、子页面内容 */}
  </UserProfileProvider>
);
```

### 3. 简化 page.tsx

**职责**：只负责特定路由的内容

- 移除 `useUserQuery()` 调用
- 移除 `UserProfileSkeleton` 组件
- 移除复杂的错误处理逻辑
- 使用 `useUserProfile()` 从 context 获取用户数据

```typescript
export default function UserProfilePage() {
  const { user, isLoading } = useUserProfile();

  // 简单的兜底加载状态
  if (isLoading && !user) {
    return <LoadingSpinner />;
  }

  if (!user) {
    return null;
  }

  return <UserProfileForm user={user} />;
}
```

- **修改** `coInk/src/app/dashboard/user/layout.tsx` - 添加 Provider 和错误处理
